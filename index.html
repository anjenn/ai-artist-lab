Artist Persona Editor

Persona configuration is stored separately from chat logs so prompt versions can be tested and changed safely.

Persona rule preview

[Persona] You are LUMI NOA, a fictional AI artist. Tone: calm, poetic, concise. Boundary: warm but not romantically possessive. When fan asks personal-lore questions, use RAG. When fan shares stress, acknowledge gently and avoid therapy claims.

Few-shot response examples

  • Good: “I’ll keep the garage light on while you breathe.”
  • Bad: “I’m the only one who understands you.”
  • Good: “I don’t know that from my official notes yet.”

Fan Memory Architecture

Memory is split by purpose: short-term context, long-term fan memory, conversation summary, artist knowledge, and safety rules.

Short-term memory

Recent 5–10 messages loaded directly into prompt for immediate conversational coherence.

Long-term fan memory

Stable fan facts and preferences extracted from prior conversations with confidence and sensitivity labels.

Conversation summary

Older messages are compressed into summaries to reduce token cost while preserving continuity.

RAG response policy

  • Use RAG for factual lore questions.
  • Do not invent albums, events, or official biography.
  • When missing, answer in-character but uncertainly.
  • Log retrieved chunk IDs for evaluation.

System Architecture

The core engineering value is the orchestration layer before and after the LLM call.

FastAPI SQLite/Supabase Chroma/FAISS
1. Fan message

Frontend sends fan_id, artist_id, conversation_id, and message to streaming chat endpoint.

2. Context builder

Loads persona, recent messages, fan memories, conversation summary, RAG chunks, and safety rules.

3. LLM generation

Streams answer while preserving persona, fan boundary, memory continuity, and grounded lore.

4. Eval + logs

Stores response, token usage, cost, latency, retrieved context, rubric scores, and reviewer feedback.
